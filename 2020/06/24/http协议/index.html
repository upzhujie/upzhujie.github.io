http协议

http协议 - 超文本传输协议， 是基于TCP/IP的应用层协议，它不涉及数据包传输，主要规定了客户端和服务器之间的通信格式

基本性质：

  • 简单：设计简单，报文容易阅读

  • 可拓展： 客户端和服务端可以拓展http headers，拓展功能

  • 无状态，有会话的： 两个http请求是不相关的，但是可以通过cookie来建立有状态的会话（本质无状态）

http是基于TCP协议的，因为TCP协议是可靠的，可以通过他的超时重传以及报文校验等等来保证数据的可靠

通过http能做什么

  • 通过Cache-Control控制缓存
  • 开放同源策略，Access-Control-Allow-Origin，使得跨域请求
  • 通过cookie建立一个带有状态的会话

关于http/1.0

在http1.0中，默认会为每一对单独的http请求都打开一个单独的TCP链接，例如在一个客户端中，同时请求了一个域名下的不同资源，他会为每一次请求都建立一个单独的TCP链接，因此这种方式是比较低效的

Q： 那么如果针对统一域名下的请求，都共享一个TCP链接，是否会提升效率呢？ - 会

关于http/1.1

在http1.1中，引入了持久链接，即TCP链接默认不关闭，可以被多个请求复用，即Connection: keep-alive(默认)，在我们建立第一次TCP链接后，TCP链接在第一次请求完成后并不会马上销毁，若有第二个对于同一域名的请求，则会复用该链接，否则，进入休眠状态，若在一定时间内发现客户端和服务端并没有请求了，则可以主动地关闭链接。

浏览器对于同一域名的请求，可以并发建立多个TCP链接，但一般数量会控制在4-8个之间

如果浏览器不对并发请求数量进行限制，会给服务器造成很大的压力，一时间会有大量请求同时发往服务端，造成堵塞

对于页面的优化方向： 减少资源请求

关于http/2

  • 帧： HTTP/2 数据通信的最小单位消息

  • 流：由帧组成数据流

多路复用

一个TCP链接中可以存在多个流，换句话说也就是可以发送多个请求，可以把每个请求当作一个流

同域名下所有通信都在单个连接上完成，该连接可以承载任意数量的双向数据流。每个数据流都以消息的形式发送，而消息又由一个或多个帧组成。多个帧之间可以乱序发送，根据帧首部的流标识可以重新组装。

结论

HTTP 报文是使用 HTTP 的关键；它们的结构简单，并且具有高可扩展性。HTTP/2 帧机制是在 HTTP/1.x 语法和底层传输协议之间增加了一个新的中间层（大概是从数据到帧和流的转化层），而没有从根本上修改它，即它是建立在经过验证的机制之上

关于https

对于http链接来说，传输的信息是铭文传输的，相当于裸奔状态，若链接被劫持，则传输的信息就会有着泄漏的风险，所以如何保证传输信息的安全，这便是需要https的缘由了

https协议，可以理解为HTTP+SSL/TSL（目前广泛TSL1.1/TSL1.2），即在TCP和HTTP层之间家了一层SSL/TSL层

加密算法

对称加密： 加密/解密全用一个公钥

非对称加密： 用公钥加密，私钥解密（安全性强，性能较低，例如RSA）

如何对http中报文进行加密呢？

  • 第一种方式：

​ 对称加密，全部使用公钥，即第一次请求服务器时，服务器返回相应的公钥，用于给客户端报文进行加密，后续在服务端解密，但是这种方式若第一次请求时被拦截，让拦截者获取了传输的公钥，则报文信息同样会泄漏

  • 第二种方式：

​ 先通过非对称加密获得公钥，后续通过对称加密对报文加密。即客户端首次请求时获取到服务器的公钥T，通过该公钥对自己客户端生成的一个公钥K进行加密，传输给服务端，服务端通过私钥解密获得K，后续的传输过程中便全用K进行加密解密。但是这种方式同样也会有风险，若劫持者在首次请求过程中，将服务端返回的公钥T替换为自己的公钥Y，发送给客户端，客户端K经过Y加密后，发送给了拦截者，拦截者通过自己对应的私钥解密，照样获得了公钥K

  • 第三种方式（安全，可靠）：

​ 引入第三方，权威证书颁发机构（CA）。首先服务端会先将公钥key1发送给机构申请证书，机构会通过他的密钥生成签名，该签名中包含了服务器网址以及其他一系列信息，然后会将该签名发送给客户端，客户端中，各大浏览器和操作系统已经维护了所有权威证书机构的名称和公钥，在客户端获得对应机构的公钥，并且根据签名规则来校验该签名是否有效。验证成功后，客户端解密出证书签名，同时客户端生成堆成加密的密钥K，通过key1对K进行加密，发往服务器，在服务端通过自己的密钥对其进行解密，获得后续传输的对称加密的K。

关于公钥和私钥的理解

让对方任意想一个3位数，并把这个数和91相乘，然后告诉我积的最后三位数，我就可以猜出对方想的是什么数字啦！比如对方想的是123，那么对方就计算出123 * 91等于11193，并把结果的末三位193告诉我。看起来，这么做似乎损失了不少信息，让我没法反推出原来的数。不过，我仍然有办法：只需要把对方告诉我的结果再乘以11，乘积的末三位就是对方刚开始想的数了。可以验证一下，193 * 11 = 2123，末三位正是对方所想的秘密数字！

其实道理很简单，91乘以11等于1001，而任何一个三位数乘以1001后，末三位显然都不变（例如123乘以1001就等于123123）。

知道原理后，我们可以构造一个定义域和值域更大的加密解密系统。比方说，任意一个数乘以400000001后，末8位都不变，而400000001 = 19801 * 20201，于是你来乘以19801，我来乘以20201，又一个加密解密不对称的系统就构造好了。
